Overview

Imagine you hired a brilliant assistant who can read your codebase, search the web, write code, run commands, and generate reports. That's Claude Code. But right now, every time you ask it to do something, you're giving instructions from scratch — like calling a contractor and explaining the job specs every single time.

Skills and agents fix that.

Analogy

Skills are like recipes — reusable instruction sets that tell Claude how to do a specific task consistently. Instead of explaining "search the codebase, write a markdown report with these sections, save it here..." every time, you write it once as a skill and invoke it with /report.

Agents are like specialized team members — each with their own expertise, tools, and context. A security reviewer agent, an explorer agent, a code improver agent. You can send them out to work in parallel, like dispatching scouts in different directions and having them report back.

Together, they transform Claude Code from a general-purpose assistant into a customized, repeatable workflow engine.

What Are Skills?

A skill is a markdown file that contains instructions for Claude. When you type /skill-name in Claude Code, it loads that file and follows the instructions — like opening a recipe card before cooking.

The Recipe Analogy

Recipe CardSkill
Recipe nameSkill name (e.g., report)
"When to make this dish"Description field — tells Claude when to use it

Step-by-step instructionsThe markdown body — what Claude should do
"Serves 4"argument-hint — what input the user provides

Where Skills Live

What Skills Are Good For

  • Repeatable workflows — "Generate a report," "Review this PR," "Fix this GitHub issue"
  • Consistent output — Every report follows the same structure, every review covers the same checklist
  • Domain knowledge — "Our API conventions," "Our error handling patterns," "Our design system"
  • Complex multi-step tasks — "Deploy to staging," "Create a new component with tests"

Creating Your First Skill

Step 1 — Create the directory

mkdir -p .claude/skills/explain-code

Step 2 — Write the SKILL.md

Create .claude/skills/explain-code/SKILL.md:

---
name: explain-code
description: Explains code with analogies and diagrams.
argument-hint: [file-or-function]
---

When explaining code, follow this structure:

1. Start with an analogy
2. Draw a diagram (ASCII art)
3. Walk through line by line
4. Highlight a gotcha

Step 3 — Use it

/explain-code pages/api/token.js

That's it. Claude reads your skill file, applies its instructions, and explains the code using analogies and diagrams — every single time, consistently.

Skill Anatomy — The SKILL.md File

Every skill has two parts: frontmatter (the metadata between --- markers) and body (the instructions in markdown).

Frontmatter Fields

FieldRequiredWhat It Does
nameNoThe name you type after /. Defaults to folder name.
descriptionRecommendedClaude reads this to decide when to auto-invoke the skill.
argument-hintNoShows [hint] in autocomplete so users know what to type.
disable-model-invocationNoWhen true, only you can invoke it. Use for destructive ops.
user-invocableNoWhen false, hides from / menu. Only Claude uses it.
allowed-toolsNoRestricts tools. E.g., Read, Grep makes it read-only.
modelNoForce sonnet, opus, or haiku.
contextNoSet to fork to run in an isolated subagent context.

Dynamic Variables

VariableExpands To
$ARGUMENTSEverything typed after /skill-name
$0, $1, $2Individual arguments by position
${CLAUDE_SESSION_ID}Unique session identifier
!`command`Output of a shell command (runs before Claude sees the prompt)

Shell Command Injection

Analogy

It's like a chef who gets fresh ingredients delivered to the prep station before starting the recipe. The !`command` syntax runs shell commands first, and Claude sees the results, not the commands.

---
name: pr-review
description: Review the current pull request
---

## PR Context (injected live)
- PR diff: !`gh pr diff`
- Changed files: !`gh pr diff --name-only`

Review the changes above for quality and security.

How Skills Get Invoked

1. User-Invoked (You Type It)

/report agents and skills
/fix-issue 123
/explain-code src/auth/login.ts

2. Auto-Invoked (Claude Triggers It)

Claude reads the description field of every skill and decides whether to load one based on your message.

SettingYou Can InvokeClaude Can Invoke
Default (no flags)YesYes
disable-model-invocation: trueYesNo
user-invocable: falseNoYes

Rule of thumb: Auto-invoke for knowledge/reference skills. Manual-only (disable-model-invocation: true) for action skills like deploy, commit, or report generation.

What Are Agents?

If skills are recipes, agents are sous-chefs — specialized assistants that work under the head chef (your main Claude Code session).

The Kitchen Analogy
KitchenClaude Code
Head chef (you + main Claude)Your main conversation
Sous-chef specializing in saucesExplore agent (fast codebase search)
Sous-chef specializing in pastryPlan agent (designing approaches)
Line cook who can do anythingGeneral-purpose agent (multi-step tasks)
Sending a sous-chef to the pantrySpawning a subagent with the Task tool
Getting the dish backSubagent returns results to main conversation

Built-In Agent Types

Explore Agent

Model: Haiku (fast)

Tools: Read-only

Like sending a scout ahead to survey the terrain. Fast codebase discovery without modification.

Plan Agent

Model: Inherited

Tools: Read-only

Like an architect surveying a building before drawing blueprints. Research and design, no implementation.

General-Purpose Agent

Model: Inherited

Tools: All tools

Like a full-stack contractor — reads blueprints, writes code, runs tests, ships results.

Bash Agent

Model: Inherited

Tools: Bash only

Isolated terminal context for running builds, tests, and git operations.

Creating Custom Agents

File Locations

ScopePathShared?
Personal~/.claude/agents/name.mdNo — just you

Example: Security Reviewer

---
name: security-reviewer
description: Security specialist. Use proactively after code changes.
tools: Read, Grep, Glob, Bash
model: sonnet
---

You are a senior security engineer reviewing code.

When invoked:
1. Run git diff to see recent changes
2. Check for OWASP Top 10 vulnerabilities
3. Look for hardcoded secrets, SQL injection, XSS
4. Report findings with severity ratings

Agent Orchestration — Parallel vs. Sequential

Parallel

Send multiple agents out at the same time when tasks are independent.

Analogy

Three scouts heading in different directions — north, south, east. They don't wait for each other.

Sequential

Chain agents when one needs the output of another.

Analogy

A relay race — the first runner finishes and passes the baton to the next.

Background agents: Press Ctrl+B to send a task to the background. Claude pre-approves permissions and runs independently, notifying you when done.

Hooks — Automating the Lifecycle

Hooks are shell commands that run at specific points in Claude Code's lifecycle. They're deterministic — they always run, unlike skills which Claude may or may not invoke.

The Assembly Line Analogy

Hooks are like quality control stations on an assembly line. At each station, a specific check happens automatically — no one needs to remember to do it.

Hook Events

EventWhen It FiresCan Block?Common Use
SessionStartSession beginsNoLoad context
PreToolUseBefore a tool executesYesValidate, protect files
PostToolUseAfter a tool succeedsNoAuto-format, notify
StopClaude finishes respondingNoVerify completeness
SubagentStartA subagent spawnsNoLog, audit
SessionEndSession terminatesNoCleanup, save state

Three Hook Types

Command

Run a shell command. Most common and performant.

"type": "command"
"command": "npx prettier --write file"

Prompt

Ask a fast AI model to make a yes/no decision.

"type": "prompt"
"prompt": "Are all tasks complete?"

Agent

Spawn a full subagent for complex verification.

"type": "agent"
"prompt": "Verify tests pass"

MCP Servers — Extending Claude's Reach

The Model Context Protocol (MCP) lets Claude connect to external services through a standardized interface.

The Plugin Analogy

MCP servers are like attachments for a power tool. Your drill (Claude) is powerful on its own, but with the right attachment, it can drive screws (query databases), bore through metal (create Jira tickets), or sand surfaces (analyze Sentry errors).

Installation

# Remote cloud service
claude mcp add --transport http github https://api.githubcopilot.com/mcp/

# Local tool
claude mcp add --transport stdio db -- npx -y @bytebase/dbhub \
  --dsn "postgresql://user:pass@localhost/db"

# Manage
claude mcp list       # List all
claude mcp remove gh  # Remove one
/mcp                  # Check status inside Claude Code

Best Practices

Skills

  • Keep SKILL.md under 500 lines — move reference material to supporting files
  • Write specific descriptions with keywords users would naturally say
  • Use disable-model-invocation: true for destructive operations
  • Include output format examples so Claude knows exactly what to produce
  • Test with simple inputs first, then iterate

Agents

  • Design focused agents — each should excel at one thing
  • Limit tool access — read-only for research, full access for implementation
  • Use background agents for long-running tasks
  • Run parallel agents when exploring independent areas

Hooks

  • Use command hooks for deterministic actions (formatting, notifications)
  • Use prompt hooks for simple decisions
  • Use agent hooks for complex verification

Start conservative: Add skills and agents as you identify repeatable patterns, not all upfront. It's better to have 3 polished skills than 15 rough ones.

Getting Started — Your First 30 Minutes

  • Minutes 0–10: Create your first skill — pick a task you do repeatedly, write SKILL.md, test with /skill-name
  • Minutes 10–20: Create a custom agent — pick a role (security, performance), write the agent file, ask Claude to use it
  • Minutes 20–30: Set up a hook — pick an automation (auto-format, notifications), add to .claude/settings.json
  • Beyond 30 min: Add MCP servers, create a /commit skill, build a /review agent
