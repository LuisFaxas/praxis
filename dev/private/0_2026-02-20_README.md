# Private Artifacts

Files placed here are part of the project context but too sensitive to commit to version control.

**When you use Praxis in your own project, add this line to your `.gitignore`:**

```
dev/private/
```

This README is visible in the template so you understand the folder's purpose. Your actual private files should never be committed.

## What Goes Here

- Server credentials, connection strings, SSH key references
- Client contracts, agreements, NDAs
- Internal notes with real names, payment details, or PII
- Infrastructure cost spreadsheets with pricing
- Any document the Source of Truth references but shouldn't be committed

## What Does NOT Go Here

- `.env` files -- those belong at your application root (your framework handles them)
- Application secrets consumed by code -- those go in `.env` or a secrets manager
- Source code -- that belongs in the application, not in dev/

## How to Reference

In `source_of_truth.md` or work orders, reference private docs by path:

```markdown
### Credentials
- Server access: see `dev/private/server_credentials.md`
- Client contract: see `dev/private/contract_v2.pdf`
```

The AI reads the reference and knows where to find the information. Git never sees the actual files.

## Why This Exists

When `dev/` is committed to a private repo (common in enterprise), everything in it becomes visible to the team. The `private/` subfolder creates a safe zone for documents that should exist in the project context but never in version control -- even in a private repo.

Think of it as the locked drawer in the filing cabinet. The office map shows where the drawer is. Only someone physically at the desk can open it.
